# skills/root-cause-analysis.md - 真因調査の標準手順

## 適用範囲

バグ調査・実装前の事前調査・修正方針決定の場面で、**CC と構築士の両者が従う標準プロセス**。
rc6・rc7・rc8 の判断ミス（思い込み実装による失敗）の再発防止が目的。

CC は事前調査フェーズの NEXT_CC_PROMPT を読んだら、本ファイルも併せて Read すること。

---

## 1. 思い込みフレーズの禁止（最重要）

以下のフレーズを **根拠なしに使ってはいけない**:

- 「自動で追従するはず」
- 「既存と同じはず」
- 「これで解決するはず」
- 「○○は動いているはず」
- 「××は変わっていないはず」

これらを使う前に、必ず実コードで確認する:
- 関連 CSS セレクタが実在するか **Grep** で確認
- 関連関数が呼ばれているか **call site** を Grep
- 関連変数が初期化されているか **定義箇所** で確認
- case 文 / switch / if-else の網羅性を実コードで確認

---

## 2. 仮説立てのルール

各仮説は **必ず実コード根拠と一緒に** 提示する:
- ファイルパス + 行番号（例: `src/main.js:1220`）
- 該当コードの引用（数行）
- なぜこれが原因と疑うか（時系列・症状との整合）

仮説の数: **3〜5 個**。多すぎると分析が浅くなる、少なすぎると見落としリスク。

---

## 3. 反論プロセスの徹底

仮説を立てたら、各仮説について **反論を試みる**:
- 「この仮説が真なら、別の症状 X も出るはず → 出ていないなら仮説誤り」
- 「この仮説が真なら、別のテスト Y が PASS しないはず → PASS しているなら仮説誤り」
- 「この仮説が真なら、別のコード Z が呼ばれないはず → 呼ばれているなら仮説誤り」

**反論できない仮説のみ** を真因候補として残す。

---

## 4. 真因確定の条件

以下の **3 つすべて** を満たして真因確定:
1. **実コード根拠**（ファイル:行番号 + 引用）
2. **症状と整合**（時系列・観測条件・再現性）
3. **既存挙動との整合**（致命バグ保護 5 件 + 既存テストすべて PASS）

3 つのうち 1 つでも欠けたら **「最有力候補」止まり** とし、構築士判断を仰ぐ。

---

## 5. 実機計測が必要なケースの見極め

以下は静的コード解析では真因確定できない:
- ブラウザ / Electron / OS の描画タイミングが絡む遅延
- レンダリングパイプラインの内部挙動
- メモリ / CPU 使用率の変動
- 並行処理の race

これらは **時刻計測ログを一時挿入** して実機で計測する:
- 挿入箇所を CC_REPORT に明記
- 削除予定を必ず宣言（最終 PR には残さない）
- 計測完了後、削除確認を CC_REPORT §9 に記載

---

## 6. 「自動追従するはず」と判断する前のチェックリスト

「○○は自動で追従するはず」と判断したくなったら、必ず:

1. 関連 CSS / DOM の現状を実コードで確認（Grep + Read）
2. 既存の意図的設計を確認（コメント / 命名から「これは意図的に空欄」を読み取る）
3. case 文の網羅性確認（switch / if-else の全パターン）
4. 「呼び忘れ」防止（新ハンドラ追加時、既存の同類関数を全部呼ぶか確認）

**このチェックを省略した結果、rc6（CSS 0 件見落とし）/ rc7（updateOperatorPane 呼び忘れ）の判断ミスが発生した。**

---

## 7. CC への反論奨励

CC が構築士の仮説や指示に対して **実コード根拠で反論できる** 場合、必ず反論する:
- 「構築士仮説 A は実コード上は否定できる（理由 X）」
- 「subagent 主張 B は CSS 仕様上誤り（理由 Y）」
- 「指示の前提 C は実コード上は成立しない（理由 Z）」

これは構築士の権威に対する反抗ではなく、**品質保証の一環**。
正直な反論ができる CC は、自動的に賢い CC である。

ただし反論には **実コード根拠が必須**、推測ベースの反論は禁止。

---

## 8. 失敗パターンと教訓（具体例）

### rc6 判断ミス
- 「AC を最小化すれば残存問題が解消する」と判断
- 見落とし: 「単画面用の CSS が 1 行も無いこと」
- 結果: minimize しても 2 画面用レイアウトが残る

### rc7 判断ミス
- 「役割を動的切替すれば CSS が自動追従するはず」と判断
- 見落とし: `[data-role="operator-solo"]` の CSS ルール 0 件 + `updateOperatorPane` 呼出忘れ
- 結果: 役割切替しても表示変わらず

### 共通教訓
- 「自動追従するはず」「既存と同じはず」は危険
- 実コード根拠を集めてから方針決定
- 事前調査フェーズを省略しない

---

## 9. 構築士・CC の責務分担

| フェーズ | 構築士 | CC |
|---|---|---|
| 仮説立て | 想定真因を NEXT_CC_PROMPT に列挙 | 仮説を実コード根拠で検証 |
| 反論 | CC の反論を尊重、根拠あれば訂正 | 実コード根拠で反論可、必ず提示 |
| 真因確定 | CC の調査結果を採点、判断 | 真因 + 補助要因 + 修正案 1〜3 を提示 |
| 修正実装 | 修正方針確定（NEXT_CC_PROMPT 別途作成）| 構築士の指示通り実装、それ以外は触らない |

---

## 10. 採点への影響

本ガイドの違反は採点に影響:
- §1（思い込みフレーズ）違反 → 実装精度 25 点中、最大 10 点減点
- §2（実コード根拠なし仮説）違反 → 報告品質 20 点中、最大 8 点減点
- §3（反論なしで仮説確定）違反 → 副作用配慮 15 点中、最大 5 点減点
- §7（実コード根拠で反論できたのに従順に従った）→ 加点対象（独自反論ボーナス）
