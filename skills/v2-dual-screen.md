# skills/v2-dual-screen.md - v2.0.0 2 画面対応の品質基準

## 適用範囲
v2.0.0 の 2 画面対応実装すべて。PIPELINE.md「v2.0.0 構築フロー」STEP 0〜7 で参照。

---

## 1. アーキテクチャ原則

### 1.1 役割分離
- **ホール側ウィンドウ**: 表示専用。設定 UI / 操作ボタン / 入力フィールドを含めない
- **PC 側ウィンドウ**: 操作専用。タイマー表示はせず、操作 UI のみ
- **状態の所在**: メインプロセス（main.js）が単一の真実源。両ウィンドウは購読側

### 1.2 単画面モード後方互換
- HDMI なし環境では v1.3.0 と完全同じ動作
- 単画面モード時は 1 ウィンドウに「表示 + 操作」を統合（既存 renderer.js の配置）
- 切替時にデータ消失・タイマーずれを起こさない

### 1.3 画面間通信の節度
- 通信は IPC（main 経由）に集約、ウィンドウ間の直接通信禁止
- 1 秒ごとの全データ転送禁止 → 差分のみ送信
- ホール側からの操作リクエスト禁止（ホール側は受信のみ）

---

## 2. 状態同期の精度基準

### 2.1 タイマー表示同期
- 両画面のタイマー表示のずれは **±100ms 以内**
- ホール側はメインプロセスから受け取った時刻情報を元にローカルで計算（ネットワーク往復削減）
- 一時停止 / 再開のレスポンスは PC 側押下から **300ms 以内** にホール側反映

### 2.2 ブラインド構造同期
- 構造変更後、ホール側に **500ms 以内** に新構造が反映される
- 編集中のフォームデータは PC 側のみで保持、保存ボタン押下までホール側に伝播させない

### 2.3 設定（背景・スライドショー等）同期
- 表示設定変更は即時（**200ms 以内**）にホール側反映
- スライドショーの画像切替タイミングは両画面で意識的に同期しない（ホール側のみで動作）

---

## 3. HDMI 抜き差し追従

### 3.1 検出
- Electron `screen` モジュールの `display-added` / `display-removed` イベントを使用
- 検出から状態切替まで **2 秒以内**

### 3.2 単画面復帰時
- ホール側ウィンドウを閉じて PC 側ウィンドウに表示を統合
- タイマー進行は中断しない（main プロセスで持続）
- 統合後は v1.3.0 のレイアウトに完全復帰

### 3.3 2 画面復帰時
- 起動時と同じモニター選択ダイアログを表示（毎回選択する仕様に従う）
- ホール側ウィンドウを再生成、状態を main から再同期

---

## 4. 起動時のモニター選択

### 4.1 選択ダイアログ
- 検出されたモニター 2 枚以上の時のみ表示
- 各モニターの解像度 / 位置 / 名称を一覧表示
- 「このモニターをホール側にする」を 1 つ選択 → 残りが PC 側
- キャンセル時は単画面モードで起動

### 4.2 選択結果の保存有無
- 起動ごとに毎回選択（記憶しない、前原さん要望）
- ただし設定ファイルに「前回の選択」を参考情報として保存可能（次回ダイアログのデフォルト選択用）

---

## 5. 禁止事項

- ホール側ウィンドウの DevTools を本番ビルドで開けるようにしない（誤操作防止）
- ホール側にマウスカーソル可視 UI を残さない（運営中の見た目を損なう）
- 既存 v1.x の状態管理（state.js / store / electron-store）を破壊する形で書き換えない
- 既存 138 テストを「無効化」「skip」「コメントアウト」しない（v2 専用テストは別ファイルで追加）
- 単画面モード時に v1.3.0 と異なる挙動を持ち込まない
- 状態同期に「ポーリング」（定期的な全取得）を使わない、必ずイベント駆動

---

## 6. テスト方針

### 6.1 既存 138 テスト
- v2 実装中も全て PASS 維持
- 単画面モードのテストは既存テストで担保（v1.3.0 の renderer.js 経路をそのまま使う）

### 6.2 v2 専用テスト
- 新規ファイル: `tests/v2-dual-screen.test.js`
- 静的解析ベース（Electron 起動なし）
- カバー対象:
  - main.js の 2 ウィンドウ生成ロジック
  - IPC ハンドラの最小性（指定の差分送信のみ）
  - HDMI 検出ハンドラの存在
  - モニター選択ダイアログの呼出条件

### 6.3 実機検証項目
- 単画面 PC で v1.3.0 と同じ動作
- HDMI 接続 → 起動 → ダイアログ → ホール側選択 → 2 画面表示
- 営業中 HDMI 抜き → 単画面に統合 → タイマー継続
- HDMI 再接続 → ダイアログ → 2 画面復帰

---

## 7. 実装上のヒント（CC 向け）

- `main.js` での 2 ウィンドウ生成は別関数に分離（`createOperatorWindow()` / `createHallWindow()`）
- 状態の真実源は **main プロセス側に置く**（既存 electron-store + 追加 state cache）
- IPC イベント名は `dual:` プレフィックスで統一（`dual:state-sync` / `dual:display-changed` 等）
- ホール側 renderer は既存 `index.html` を流用、フラグ（querystring or process.argv）で操作 UI を hidden 化
- 詳細は STEP ごとの NEXT_CC_PROMPT.md で個別指示

---

## 8. 致命バグ保護への接続点（必読）

v2 実装で**特に注意が必要**な v1.x 致命バグ保護:

- `resetBlindProgressOnly`（C.2.7-A）: 2 画面化で reset 経路が増えても**この関数経由を維持**
- `timerState` destructure 除外（C.2.7-D）: payload 同期で timerState が race しないよう経路設計
- `ensureEditorEditableState` 4 重防御（C.1-A2）: PC 側のみで動作するため 2 画面化の影響なしと予想、ただし要確認
- AudioContext resume（C.1.7）: 音はホール側で鳴らす想定、PC 側操作 → ホール側で resume する経路設計
- runtime 永続化 8 箇所（C.1.8）: 操作は PC 側、永続化は main 経由なので変更不要と予想

各 STEP で「この致命バグ保護への影響なし」を確認してから次へ進む。
