# 構想 / ロードマップ — マルチトーナメント4分割表示モード（poker-clock）

> 立案: CC構築士2（Cowork版）× 前原 壁打ち 2026-06-24。**これは設計の正典（前原確定事項＋ガードレール＋段階）**。
> 実装は CC が本書を読んで自走（`feedback_cowork_roadmap_cc_selfrun`）。危険箇所（下記ガードレール）に触れたら escalate。

## 1. 目的 / 使う場面
- **大型大会**で、**映画スクリーンのような巨大モニター**に、複数のトーナメント時計を**同時に並べて見せる**ための専用表示モード。
- 顧客要望発。店舗の通常営業はPC2台運用のため対象外（このモードは大型大会用）。

## 2. 前原 確定事項（壁打ちで決定・仕様の土台）
1. **4分割（2×2）専用**。2分割・3分割は作らない。3大会のときは4分割のまま1区画を空きにする。
2. **各区画 = 独立した“普段どおりの完全な時計”**。保存済みトーナメントから選ぶ。**独立**＝開始時刻・ブラインド構造・一時停止・レベル進行、すべて区画ごとにバラバラに動く。
3. **このモードで無効化するのは3つだけ**＝休憩スライドショー / 音 / テロップ（marquee）。**それ以外の表示情報は全部出す**。
4. **操作は手元PC1台**に「4区画ぶんのミニ操作盤」を並べる：区画ごとに スタート（独立した開始時刻／カウントダウン可）・一時停止・レベル送り/戻し・リセット。巨大スクリーンは**見せるだけ**。
5. **HDMI拡張（extend）が本命**：手元PC＝操作盤／巨大スクリーン＝2×2グリッド。ノートPC本体画面を操作盤にすれば追加機材不要。
6. **複製（mirror）の保険＝キーボード操作を搭載**：グリッド全画面＋ショートカットで、操作UIを画面に出さずに1台運用できる経路も用意。
7. **空き区画（トーナメント未割当）に映すもの**は操作者が選べる：無地（黒）／ロゴ／任意画像1枚（告知フライヤー等）／固定テキスト。
8. 巨大スクリーン前提のため**1/4でも十分大きい** → 副次情報を縮める可変ルールは**不要**（各区画フル表示でよい）。

## 3. 各区画に表示する情報（普段の時計の全項目・暫定リスト）
> ⚠️ 実装前に **CC がソースから“表示項目の完全リスト”を機械的に洗い出して確定**する（人力の見落とし防止）。以下は壁打ち時点の棚卸し。

- 左：ロゴ / presented by / 配当表(PAYOUTS) / 総プール額（¥・GTD注記）
- 中央：レベル番号 / 大タイマー（状態ラベル：ブレイク中・スタートまで・一時停止中・終了）/ 現ブラインド(SB-BB-アンティ) / 次ブラインド
- 右：NEXT BREAK IN（または TOTAL GAME TIME）/ 平均スタック / 参加人数 / REENTRY数 / ADDON数 / 特別スタック（早期着席特典）
- **無効化**：休憩スライドショー・音・テロップ の3つのみ。

## 4. 設計原則 / ガードレール（安全網・厳守）
- **① 既存の1トーナメント表示に手を入れない**：新モードは**独立したエンジン**（独立した時計の状態を必要数だけ持てる作り）として実装する。既存の「状態1個」の単一モード経路と**致命バグ保護5件**（resetBlindProgressOnly / timerState destructure除外 / ensureEditorEditableState 4重防御 / AudioContext resume / runtime永続化8箇所）には**非接触**を最優先。
- **② 性能ゲート**：巨大解像度で**4つの時計を同時に60fps級で滑らかに**動かせること。カクつくならアプローチを見直す（早期スパイクで検証、下記 Phase 0）。
- **③ reuse-vs-rebuild を先に判断**：各区画は「普段の時計まるごと」なので、既存の時計描画・時計ロジックを**どこまで再利用**できるか（部品化して4回使う／新規に軽量描画する）を Phase 0 で決める。安易な全面書き直しは既存の積み上げ（多数のバグ修正）を失うリスク、singleton前提の流用は状態衝突リスク。両にらみで判断。
- **④ dual-screen / window管理は慎重**：extend時の会場ウィンドウ配置・HDMI抜き差し・operator-solo復帰など既存の会場表示ロジックに触れる可能性。**認証/SQLは無関係**（ローカルElectronアプリ）だが、状態同期(dual-sync)は静かに壊れやすいので、確信が持てなければ escalate。
- **⑤ 段階を守る**：一気に作らず下記フェーズで刻む。各フェーズ完了で前原の実機確認（巨大スクリーン相当があれば理想、無ければ通常モニターで代替）。

## 5. 段階（フェーズ）
> CCが案件番号を採番。各フェーズは独立に review/実機確認できる粒度で。

### Phase 0 — 調査・設計検証（本番挙動を変えない）
- 現行の 状態(state.js) / 時計(timer.js) / 会場同期(dual-sync.js) / 描画(renderer) の構造を調査。
- **表示項目の完全リスト**を確定（§3 の裏取り）。
- **独立エンジン方針を決定**：N個の独立した時計状態をどう持つか（状態のインスタンス化／新規軽量実装）、既存単一モードと致命バグ保護5件に非接触でいける設計かを技術的に示す。
- **性能スパイク**：4区画フル描画が滑らかに動く見込みを実測 or プロトタイプで確認。
- 出力＝設計/技術判断のレポート（reuse-vs-rebuild結論・リスク・Phase1の具体設計）。**GО/見直しの分岐**を明示。

### Phase 1 — 骨格（2×2グリッド＋独立時計）
- 「マルチ表示モード」への入口を新設（既存モードと切替、戻れる）。
- 会場側＝2×2グリッドに、保存済みトーナメントから割り当てた**最大4つの独立した完全な時計**を描画（休憩スライドショー/音/テロップは無効）。
- 手元PC側＝**4区画ぶんの操作盤**（各区画：スタート/一時停止/レベル送り・戻し/リセット）。extendモデル。
- 空き区画フィラー：最低 **無地／ロゴ** を選択可。
- 独立に動く（片方だけ一時停止・レベル送りが他区画に影響しない）ことをテストで担保。

### Phase 2 — 独立開始時刻＋キーボード保険＋フィラー拡充
- 区画ごとの**開始時刻設定／スタートまでカウントダウン**（既存 PRE_START 相当を区画独立で）。
- **キーボード操作フォールバック**（mirror用）：操作対象区画の選択＋各操作のショートカット。全画面グリッドで操作UIを出さずに1台運用。
- 空き区画フィラー拡充：**任意画像1枚／固定テキスト**。
- 実機FB対応: 2b 操作対象バッジ / 2c オレンジ太枠＋±30秒 / 2d 単一モード操作パリティ / 2e 停電・クラッシュ復帰（`userData/multi-session.json`・停電時点PAUSED復元・正常終了で削除）。
- **2f 復元方式の選択**（実機FB第5弾・2026-07-08）: 復元ダイアログで「そこから再開（時計は進めない）」/「経過を反映して復元（時計を進める）」を選択制に。※2e 壁打ちの「壁時計継続は不採用」を前原指示で選択制へ上書き。どちらも復元直後は一時停止（経過反映で全レベルを終えた区画は終了として復元）。

### Phase 3 — 仕上げ・堅牢化・配信
- 性能ハードニング（巨大解像度）・エッジケース（区画の割当変更、途中でトーナメント差し替え、HDMI抜き差し、終了区画の扱い）。
- 実機確認（可能なら巨大スクリーン/プロジェクター）→ 配信（新バージョン）。

## 6. 各フェーズ共通の DoD 骨子
- 致命バグ保護5件 **非接触**（or 影響あり＋対処）を report 明記。
- 既存テスト全 PASS 維持＋新規テスト（独立性・区画割当・フィラー・キーボード操作 等）。
- 性能所見（該当フェーズ）を report 明記。
- PROGRESS.md 更新。並列 sub-agent 最大3体。
- **escalate 条件**：既存 state/dual-sync のコア改変が避けられない／致命バグ保護5件に触れる／reuse方針で単一モードを壊しかねない、と判断したら止めて Cowork へ。

## 7. 未決・将来検討（今回スコープ外の候補）
- 巨大スクリーンの解像度・アスペクト比の実値（4Kか、超横長か）→ 性能・レイアウト設計に効く。分かれば前原から共有。
- 終了/FINISHED になった区画を自動でフィラー化するか、手動か。
- 区画割当のプリセット保存（イベントごとの4区画セットを保存・呼出）。
